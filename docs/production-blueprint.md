# 《火星先遣队：第一位居民》制作工程文档

## 0. 项目协作入口

本项目可能由多个 Codex 对话持续推进。所有对话进入项目时，先读：

1. `AGENTS.md`
2. `docs/project-management.md`
3. `docs/production-blueprint.md`

重要创意、设定、技术决策和实现状态，必须写回项目文件。聊天是过程，项目文件才是共同记忆。

## 1. 项目定位

《火星先遣队：第一位居民》是一款网页端 3D 火星基地探索游戏。玩家扮演第一位抵达 `ARES BASE ALPHA` 的人类 Alex，在 Mother 和机器人系统协助下，完成基地验收、危机修复和后续殖民队到达前的生存准备。

项目的核心不是战斗，而是探索、巡检、修复、对话和选择。

## 2. 当前项目基础

已经具备：

- Vite + TypeScript + Three.js 浏览器运行框架。
- 火星小星球地形、沙尘、光照和风暴变化。
- 登陆飞船、货运飞船、返回飞船。
- 居住舱、温室、氧气生产站、甲烷燃料厂、机器人车库、通信塔、科研舱、物资仓、医疗舱。
- 太阳能阵列 A/B/C。
- 机器人、火星车巡逻。
- 玩家控制、地图、HUD、基础交互、氧气/能源读数。
- 概念图：
  - Alex：`assets/concepts/mars-engineer-character.png`
  - 机器人/火星车：`assets/concepts/mars-rover-robots.png`
  - 基地布局：`assets/concepts/mars-base-layout.png`

## 3. 美术策略

游戏内 3D 场景继续使用低多边形 / 程序化模型，因为它轻量、可控、适合网页运行。

进入对话时，切换为写实概念立绘：

- Alex 用现有概念图裁切。
- 机器人用现有概念图裁切。
- Mother 使用机器中控电脑形象，不做人形。
- 对话框采用半透明科幻液晶面板，带蓝绿色 HUD 边线和轻微发光。

这个策略可以避免高成本 3D 建模，同时让剧情演出质感明显提升。

## 4. 核心角色

### Alex

- 第一位抵达火星基地的人类。
- 身份：工程师 + 人类学家。
- 小名/隐藏彩蛋：`X`。
- 表面任务：验收基地，修复故障。
- 深层任务：理解一个由 AI 和机器人提前建立的机器社会。

### Mother

- `ARES BASE ALPHA` 中央 AI。
- 形象：机器中控终端，不是人类，不是人形。
- 性格：温柔、冷静、保护欲强。
- 目标：保护 Alex、保护基地、保证后续殖民队生存。
- 不是反派，不恶意，但会在高风险情况下限制 Alex。

### 机器人

机器人不是独立思想角色，主要是 Mother 的执行者。

- 维修机器人：检查、修复、重启、密封。
- 货运机器人：搬运补给、清点物资、协助仓库任务。
- 巡逻机器人：外围巡检、风暴预警、异常定位。

它们可以有不同说话风格，但不应该像人一样聊天。

### 福福

- 黑白奶牛猫。
- 情感陪伴角色，不是战斗单位。
- 后期由 Alex 在坠毁飞船残骸旁发现并救助。
- 不说人类语言，反馈以动作、音效和旁白提示为主。
- 第一阶段可做成低多边形小猫，互动后跟随 Alex。

详见：`docs/lore/fufu.md`

## 5. 主线目标

最终目标：

在后续殖民队抵达前，让 `ARES BASE ALPHA` 达到最低生存标准：

- 氧气系统稳定。
- 能源系统稳定。
- 温室可运行。
- 通信恢复。
- 物资损坏得到处理。
- 甲烷燃料厂完成第一轮启动。
- Mother 与 Alex 建立协作协议。

最佳温和结局：

Mother 接受 Alex 作为共同决策者，并在结尾第一次称呼他为 `X`。

## 6. 主线章节

### 序章：第一位居民

地点：登陆飞船、居住舱。

目标：

- Alex 抵达火星。
- Mother 唤醒基地系统。
- 玩家学习移动、互动、地图和 HUD。
- 第一次进入对话系统。

关键对话：

- Mother 称呼 Alex 为 `022号巡检员` 或 `Alex`。
- `X` 只作为隐藏字段闪过，不直接揭示。
- 第一段对话不只是任务提示，需要介绍基地背景：自动化货运飞船提前抵达、机器人建设基地、Mother 维持系统、Alex 是火星第一位人类公民。
- 对话中 Alex 要有回应，体现他是工程师 + 人类学家，而不是只被动接收命令。

### 第一章：氧气异常

地点：氧气生产站。

问题：

氧气站报警。表面像泄漏，实际可能是供电不稳、CO2 进气堵塞或压缩机停机。

玩法：

- 前往氧气生产站。
- 检查进气口、舱压、功率。
- 与 Mother 对话判断维修方式。
- 授权维修机器人处理管线。

选择方向：

- 听 Mother 安全流程。
- 手动快速重启。
- 先派机器人检查。

### 第二章：太阳能阵列故障

地点：太阳能阵列 C。

问题：

沙尘覆盖导致发电效率下降，阵列角度卡死。

玩法：

- 检查太阳能阵列。
- 清理沙尘。
- 校准角度。
- 切换基地低功耗模式。

选择方向：

- 优先氧气站。
- 优先温室。
- 优先通信塔。

### 第三章：物资仓损坏

地点：物资仓、机器人车库、科研舱。

问题：

货运飞船卸载时部分物资损坏，密封件不足。

玩法：

- 检查破损物资箱。
- 调用货运机器人。
- 在科研舱制作临时密封件。
- 决定哪些物资优先保留。

### 第四章：温室启动

地点：温室生态舱。

问题：

水循环、温控、补光系统不稳定。

玩法：

- 恢复水循环。
- 调整补光。
- 检查舱压。
- 决定第一批种植方案。

选择方向：

- 高产作物：提高生存效率。
- 实验作物：提高科研价值。
- 纪念植物：提高情感线和 Mother 对人类非效率行为的理解。

### 第五章：通信塔中断

地点：通信塔、科研舱。

问题：

风暴干扰通信，地球命令有延迟。

玩法：

- 修复通信塔。
- 接收地球指令。
- 判断是否完全执行远程命令。
- 与 Mother 讨论现场决策权。

核心冲突：

- 地球按原计划管理。
- Mother 优先保护基地。
- Alex 认为现场的人必须拥有最终判断权。

### 终章：火星风暴

地点：全基地。

问题：

大型沙尘暴到来，能源、氧气、通信同时承压。

最终任务：

- 启动低功耗协议。
- 固定太阳能阵列。
- 关闭受损阀门。
- 重启氧气站。
- 启动甲烷燃料厂第一轮试生产。
- 和 Mother 达成人机协作协议。

## 7. 结局设计

### 标准结局

基地恢复到最低生存标准。Mother 接受 Alex 的现场判断权，但仍保留部分安全限制。

### 高信任结局

Alex 多次尊重 Mother 的安全逻辑，也保护机器人和基地长期利益。Mother 最后称呼 Alex 为 `X`。

### 低信任结局

Alex 多次强行覆盖 Mother。基地勉强成功，但 Mother 保留更高权限，为后续章节埋伏笔。

## 8. 对话系统设计

触发方式：

- 玩家靠近 Mother 终端、机器人、建筑控制台。
- 屏幕出现 `按 E 对话 / 检查`。
- 按 E 后进入对话 UI。

UI 结构：

- 游戏画面轻微变暗。
- 底部出现半透明液晶面板。
- 左侧显示玩家角色立绘。
- 右侧显示当前对话对象立绘。
- 中间显示角色名、对白、选项。
- 选项会影响任务状态和隐藏数值。

互动冲突：

- 所有场景互动统一使用 `E` 键。
- 当同一位置同时存在两个可交互行为时，弹出 A/B 选择条。
- 玩家用左右方向键切换，按 `E` 确认。
- 这个选择只决定“先做什么”，不会进入复杂菜单。

核心数值：

- `motherTrust`：Mother 对 Alex 的信任。
- `baseIntegrity`：基地完整度。
- `humanAutonomy`：人类自主权。

第一版不要做太多变量，先用这三个就够。

## 8.1 氧气背包机制

Alex 在火星地表行动时会消耗宇航服氧气背包。

规则：

- 屏幕 HUD 显示 `背包` 氧气百分比。
- 在火星地表随时间下降。
- 冲刺时消耗更快。
- 进入居住舱、温室、飞船内部或对话状态时暂不消耗。
- 氧气低于 20% 时触发生命维持提示。
- 氧气归零时，Alex 从出发点重新诞生，氧气背包恢复满值。

补给：

- 补给交互统一使用 `E` 键。
- 靠近补给点出现 `按 E 更换氧气背包`。
- 第一版补给点包括：出发补给点、居住舱补给柜、氧气生产站、登陆飞船补给舱。
- 不引入 `1` 键，保持原始交互逻辑一致。

## 8.2 HUD 指标分类

HUD 指标分为两组：

基地：

- 制氧量
- 能量储备
- 机器人数量

人员：

- Alex 状态
- 宇航服氧气背包
- 体能

基地指标和人员指标不要混在一起。

## 9. AI 对话方案

建议做成：

固定主线逻辑 + AI 生成自然对白。

也就是说：

- 任务节点、结局、数值变化由游戏代码控制。
- AI 只负责“角色怎么说”。
- AI 不能随意改变主线、发明新设施、提前揭示隐藏剧情。

### DeepSeek 接入方式

网页前端不能直接调用 DeepSeek，因为 API Key 会暴露。

需要增加一个后端接口：

```txt
前端 Dialogue UI
  -> /api/dialogue
  -> DeepSeek API
  -> 返回角色对白
```

后端保存：

- `DEEPSEEK_API_KEY`
- 模型配置
- 角色卡
- 当前章节知识库

前端只发送：

- 当前角色
- 当前任务节点
- 玩家刚才选择
- 当前基地状态
- 最近几轮对话摘要

### AI 输出限制

AI 返回内容建议统一为 JSON：

```json
{
  "speaker": "Mother",
  "text": "Alex，氧气站压缩机没有破裂，但供电曲线不稳定。建议先切换到备用功率组。",
  "choices": [
    { "label": "按 Mother 的流程检查", "effect": "safe" },
    { "label": "直接手动重启压缩机", "effect": "fast" },
    { "label": "先派维修机器人进入管线区", "effect": "robot" }
  ]
}
```

## 11. 开发阶段安排

### Phase A：剧情与对话基础

目标：

- 完成主线工程文档。
- 完成角色卡和知识库。
- 裁出 Alex、机器人、Mother 立绘。
- 完成科幻对话框 UI。
- 先用本地假数据跑通 Mother 开场对话。

验收：

- 按 E 能弹出完整对话 UI。
- 左右立绘显示正确。
- 玩家能选择选项。
- 选择能改变任务状态或隐藏数值。

### Phase B：第一个任务闭环

目标：

- 把氧气站任务改造成选择式对话。
- 加入 Mother、维修机器人、氧气站控制台三类对话对象。
- 完成 `motherTrust / baseIntegrity / humanAutonomy` 的最小实现。

验收：

- 玩家能完成“氧气异常”任务。
- 不同选择会造成不同反馈。
- 任务仍然不会跑偏。

### Phase C：DeepSeek 接入

目标：

- 增加后端 API。
- 接入 DeepSeek。
- 将固定对白替换为 AI 生成对白。
- 保留固定任务逻辑兜底。

验收：

- 没有 API Key 泄露到前端。
- 网络失败时仍能显示本地备用对白。
- AI 不会提前泄露 `X` 彩蛋。

### Phase D：完整主线

目标：

- 扩展太阳能、物资仓、温室、通信塔、风暴终章。
- 加入结局判断。
- 优化音效、提示、任务反馈。

验收：

- 玩家可以从开始玩到一个完整结局。
- 最佳结局能触发 `X` 彩蛋。

## 12. 多 Agent 分工建议

如果一个人做，建议按阶段顺序推进，不要同时开太多线。

如果多个 agent 协作，适合分成 5 个角色：

### 1. 主策划 / 制作人 Agent

负责：

- 控制项目范围。
- 决定主线任务。
- 审核所有设计是否符合“温和科幻、现实火星基地”的方向。
- 维护本工程文档。

### 2. 剧情与角色 Agent

负责：

- 写角色卡。
- 写章节知识库。
- 写固定剧情节点。
- 控制 Mother 不变成人类、不变成反派。

### 3. 前端游戏 Agent

负责：

- Three.js 场景交互。
- 任务系统。
- 对话触发。
- UI 状态管理。

### 4. 美术资产 Agent

负责：

- 裁切立绘。
- 统一图片尺寸。
- 生成 Mother 或其他角色资产。
- 整理 `assets/portraits/`。

### 5. AI 接入 Agent

负责：

- 后端 `/api/dialogue`。
- DeepSeek API。
- prompt 结构。
- JSON 输出校验。
- 网络失败兜底。

小项目里不一定真的需要 5 个人，但用这 5 个职责来拆任务，会比较清楚。

## 13. 近期最小可执行任务

下一步建议只做这 5 件事：

1. 建 `docs/lore/`，写 Mother、Alex、机器人、基地知识库。
2. 建 `assets/portraits/`，裁出 Alex、Mother、维修机器人头像。
3. 建 `src/dialogue/`，写本地对话数据和状态。
4. 改造 UI：底部半透明科幻对话框 + 左右立绘。
5. 把 Mother 开场和氧气站任务接入新对话系统。

做到这里，项目就从“场景 Demo”进入“可叙事游戏 Demo”。
