## 1. 项目定位

《火星先遣队：第一位居民》是一款网页端 3D 火星基地探索游戏。玩家扮演第一位抵达阿瑞斯阿尔法基地（ARES BASE ALPHA）的人类亚历克斯（Alex），在基地中央 AI 史蒂夫（Steve）和机器人系统协助下，完成基地验收、危机修复和后续殖民队到达前的生存准备。

项目的核心不是战斗，而是探索、巡检、修复、对话和选择。

### 2.1 多人模式 v1

当前多人模式定位为“免注册公共火星大厅”：

- 所有玩家默认进入同一个公共房间：`public-mars-alpha`。
- 不注册、不登录；浏览器首次进入时生成匿名 `playerId`，保存在 `localStorage`。
- 每个玩家都会生成自己的宇航员，其他玩家可在 3D 场景中看到该宇航员的位置、朝向、移动状态、飞行状态和室内/室外状态。
- 左侧人员 HUD 显示实时在线人数，人数为本地玩家加当前房间远程玩家。
- 主线和支线任务内容完全相同，但进度各自独立；一个玩家完成任务不会推进其他玩家的任务。
- 远程玩家不可碰撞、不可互动，不参与任务触发。
- IP 不作为身份主键，只能作为服务器日志、限流或防刷的辅助信息。
- 多人同步只是增强层；WebSocket 服务不可用时，游戏自动降级为单人体验。

## 3. 美术策略

游戏内 3D 场景继续使用低多边形 / 程序化模型，因为它轻量、可控、适合网页运行。

进入对话时，切换为写实概念立绘：

- 亚历克斯用现有概念图裁切。
- 机器人用现有概念图裁切。
- 史蒂夫使用机器中控电脑形象，不做人形。
- 对话框采用半透明科幻液晶面板，带蓝绿色 HUD 边线和轻微发光。

这个策略可以避免高成本 3D 建模，同时让剧情演出质感明显提升。

## 4. 核心角色

### 亚历克斯（Alex）

- 第一位抵达火星基地的人类。
- 身份：工程师 + 人类学家。
- 小名/隐藏彩蛋：`X`。
- 表面任务：验收基地，修复故障。
- 深层任务：理解一个由 AI 和机器人提前建立的机器社会。

### 史蒂夫（Steve）

- 阿瑞斯阿尔法基地中央 AI。
- 形象：机器中控终端，不是人类，不是人形。
- 性格：直接、挑剔、重视聚焦、端到端体验和系统品质。
- 目标：保护亚历克斯、保护基地、保证后续殖民队生存，并让基地从“能跑”变成“足够好”。
- 不是反派，不恶意，但会在高风险或低质量决策下限制亚历克斯。

### 机器人

机器人不是独立思想角色，主要是史蒂夫的执行者。

- 维修机器人：检查、修复、重启、密封。
- 货运机器人：搬运补给、清点物资、协助仓库任务。
- 巡逻机器人：外围巡检、风暴预警、异常定位。

它们可以有不同说话风格，但不应该像人一样聊天。

### 福福

- 黑白奶牛猫。
- 情感陪伴角色，不是战斗单位。
- 后期由亚历克斯在坠毁飞船残骸旁发现并救助。
- 不说人类语言，反馈以动作、音效和旁白提示为主。
- 第一阶段可做成低多边形小猫，互动后跟随亚历克斯。

详见：`docs/lore/fufu.md`

### 埃隆（Elon）

- 实验性 AI 智能体 / 工程顾问人格。
- 简体中文版显示为“埃隆”，英文版显示为 `Elon`；只是同名，不指向现实中的埃隆·马斯克本人，也不是数字复活。
- 位置放在 `03 飞船 返回飞船` 升降平台到达顶端后的固定高空廊道，靠近飞船舱门处；玩家需要先修复 3 号飞船升降梯才能见到它。
- `03 飞船 返回飞船` 内舱不可进入，埃隆只在高空廊道靠近舱门处与玩家互动。
- 当前已接入游戏：03 飞船升降梯修复支线、高空廊道处的埃隆、固定对白池。开放式 AI 后端暂未接入。

### 6.0 任务扩展入口

后续扩展采用 3 个主线任务包 + 3 个支线任务包，详细步骤见：

- `docs/tasks/mission-expansion-v2.md`

特殊角色解锁支线见：

- `docs/tasks/elon-ship-03-elevator-sidequest.md`

任务扩展原则：

- 不推翻当前 1.0 已实现内容，而是把现有氧气站、太阳能阵列 C、机器人车库扩成第一条主线任务的一部分。
- 主线围绕生命支持、温室启动、通信延迟、风暴协议和史蒂夫协作权展开。
- 支线围绕福福、物资仓错位货箱、巡逻机器人外围异常展开。
- 埃隆支线围绕 3 号飞船升降梯修复展开，是后续开放对话角色的解锁任务，不强制阻断主线。
- 每个任务都要服务“人类进入机器先行建设的火星基地”这个主题。

### 序章：第一位居民

地点：登陆飞船、居住舱。

目标：

- 亚历克斯抵达火星。
- 史蒂夫唤醒基地系统。
- 玩家学习移动、互动、地图和 HUD。
- 第一次进入对话系统。

关键对话：

- 史蒂夫称呼亚历克斯为 `022号巡检员`、`亚历克斯` 或 `火星第一位人类公民`。
- `X` 只作为隐藏字段闪过，不直接揭示。
- 第一段对话不只是任务提示，需要介绍基地背景：自动化货运飞船提前抵达、机器人建设基地、史蒂夫维持系统、亚历克斯是火星第一位人类公民。
- 对话中亚历克斯要有回应，体现他是工程师 + 人类学家，而不是只被动接收命令。

序章呼叫流程：

- 亚历克斯进入基地后，先由宇航服 AI 操作系统收到外部信息，而不是直接弹出角色对话。
- 信息栏显示：“收到史蒂夫的一条信息，是否接收？”
- 信息栏提供两个系统选项：`稍后接收` 和 `接收`。桌面端 `Q` 对应稍后接收，`E` 对应接收；触摸端直接点击选项。默认高亮必须落在 `接收` 上，避免误导玩家错过关键通信。
- 信息栏出现后 `30` 秒内没有选择，会自动收起，并在 `1` 分钟后重新发送。
- 选择 `稍后接收` 时，不进入对话栏，史蒂夫延迟 `2` 分钟后重试。
- 选择 `接收` 后，才进入对话栏，开始史蒂夫与亚历克斯的双向通信。
- 信息栏只承载“收到的信息、系统确认、任务提示、风险提醒”等单向内容；对话栏只用于线上通话、面对面交流、机器人问答和有角色回应的双向交流。

### 第一章：氧气异常

地点：氧气生产站。

问题：

氧气站报警。表面像泄漏，实际可能是供电不稳、CO2 进气堵塞或压缩机停机。

玩法：

- 前往氧气生产站。
- 检查进气口、舱压、功率。
- 与史蒂夫对话判断维修方式。
- 授权维修机器人处理管线。

选择方向：

- 听史蒂夫安全流程。
- 手动快速重启。
- 先派机器人检查。

### 第二章：太阳能阵列故障

地点：太阳能阵列 C。

问题：

沙尘覆盖导致发电效率下降，阵列角度卡死。

玩法：

- 检查太阳能阵列。
- 清理沙尘。
- 校准角度。
- 切换基地低功耗模式。

选择方向：

- 优先氧气站。
- 优先温室。
- 优先通信塔。

### 第三章：物资仓损坏

地点：物资仓、机器人车库、科研舱。

问题：

货运飞船卸载时部分物资损坏，密封件不足。

玩法：

- 检查破损物资箱。
- 调用货运机器人。
- 在科研舱制作临时密封件。
- 决定哪些物资优先保留。

### 第四章：温室启动

地点：温室生态舱。

问题：

水循环、温控、补光系统不稳定。

玩法：

- 恢复水循环。
- 调整补光。
- 检查舱压。
- 决定第一批种植方案。

选择方向：

- 高产作物：提高生存效率。
- 实验作物：提高科研价值。
- 纪念植物：提高情感线和史蒂夫对人类非效率行为的理解。

### 第五章：通信塔中断

地点：通信塔、科研舱。

问题：

风暴干扰通信，地球命令有延迟。

玩法：

- 修复通信塔。
- 接收地球指令。
- 判断是否完全执行远程命令。
- 与史蒂夫讨论现场决策权。
- 通信塔校准后，雷达和任务提示显示 `ARES Starlink-M` 轨道链路状态；该系统提升基地通信冗余，但不取消地火通信延迟。
- 当前星链表现为固定轨道星座：天空中直接存在 `5` 组星链，每组 `7` 颗，在不同轨道面循环飞过火星上空；不再设计按火星日定时发射或释放卫星的任务。

核心冲突：

- 地球按原计划管理。
- 史蒂夫优先保护基地和系统质量。
- 亚历克斯认为现场的人必须拥有最终判断权。
- `ARES Starlink-M` 是工程冗余，不是万能即时通信；它能改善火星本地覆盖和轨道中继，不能让地球命令实时到达。

### 终章：火星风暴

地点：全基地。

问题：

大型沙尘暴到来，能源、氧气、通信同时承压。

最终任务：

- 启动低功耗协议。
- 固定太阳能阵列。
- 关闭受损阀门。
- 重启氧气站。
- 启动甲烷燃料厂第一轮试生产。
- 和史蒂夫达成人机协作协议。

## 7. 结局设计

### 标准结局

基地恢复到最低生存标准。史蒂夫接受亚历克斯的现场判断权，但仍保留部分安全限制。

### 高信任结局

亚历克斯多次尊重史蒂夫的安全逻辑，也保护机器人和基地长期利益。史蒂夫最后称呼亚历克斯为 `X`。

### 低信任结局

亚历克斯多次强行覆盖史蒂夫。基地勉强成功，但史蒂夫保留更高权限，为后续章节埋伏笔。

## 8. 对话系统设计

触发方式：

- 玩家靠近史蒂夫终端、机器人、建筑控制台。
- 屏幕出现 `按 E 对话 / 检查`。
- 按 E 后进入对话 UI。

UI 结构：

- 游戏画面轻微变暗。
- 底部出现半透明液晶面板。
- 左侧显示玩家角色立绘。
- 右侧显示当前对话对象立绘。
- 中间显示角色名、对白、选项。
- 选项会影响任务状态和隐藏数值。

信息栏 / 对话栏边界：

- 信息栏：宇航服 AI 操作系统、基地系统、任务系统或风险系统收到/显示的单向信息；不写“互动”，不显示角色立绘，不承担剧情表演。
- 对话栏：玩家已经接通通信或靠近角色后发生的双向交流；可以显示角色名、立绘、对白和选择。
- 呼叫、短信、任务提醒、系统确认先进入信息栏；只有玩家接收、接听、靠近并开始交流后，才进入对话栏。

互动冲突：

- 桌面端所有场景互动和对话推进统一优先使用 `E` 键。
- 触摸端靠近对象后直接显示可点击互动菜单，不要求模拟 `E` 键。
- 当同一位置同时存在多个可交互行为时，按优先级只显示最重要的前两个选项。
- 桌面端两个并列选项时，`Q` 直接执行左侧，`E` 直接执行右侧；触摸端直接点击对应按钮。
- 史蒂夫呼叫出现时，先作为宇航服 AI 操作系统的信息栏出现，内容为“收到史蒂夫的一条信息，是否接收？”。信息栏提供两个系统选项：`拒绝`（Q）和 `接收`（E），默认高亮 `接收`；选择接收后，才进入真正的对话栏。
- 互动优先级：史蒂夫呼叫、进出建筑/升降梯、当前任务、机器人/福福、氧气补给。
- 这个选择只决定“先做什么”，不会进入复杂菜单。

雷达显示：

- 游戏开始后雷达直接显示已知设施和未知对象，不再使用 `O` 键开启未知点。
- 未知生物、未知残骸、黑色方碑和埃隆未被发现或对话前在雷达上统一显示为 `?`；首次发现或对话后移除 `?`，改用对应分类圆点并显示真实名称。
- 当前任务目标使用黄色圆点，并发出一圈圈放大渐隐的黄色雷达光圈；氧气补给点使用绿色圆点；金币使用较小的 `$` 金币样式且不发光。
- 雷达最外圈显示 `N/E/S/W` 方位环，随玩家视线左右旋转；坐标下方显示 `HDG 000°` 格式的朝向读数，帮助玩家按方位描述路线。

核心数值：

- `motherTrust`：史蒂夫对亚历克斯的信任。代码变量名暂时保留旧名，显示和设定统一称为史蒂夫。
- `baseIntegrity`：基地完整度。
- `humanAutonomy`：人类自主权。

第一版不要做太多变量，先用这三个就够。

## 8.1 氧气背包机制

亚历克斯在火星地表行动时会消耗宇航服氧气背包。

规则：

- 屏幕 HUD 显示 `背包` 氧气百分比。
- 在火星地表随时间下降。
- 冲刺时消耗更快。
- 进入居住舱、温室、飞船内部或对话状态时暂不消耗。
- 氧气低于 20% 时触发生命维持提示。
- 氧气低于 35% 时，雷达把可补氧地点标为绿色无外发光圆点，帮助玩家寻找最近补氧点。
- 氧气归零时，亚历克斯从出发点重新诞生，氧气背包恢复满值。

补给：

- 补给交互统一使用 `E` 键。
- 靠近补给点出现 `按 E 更换氧气背包`。
- 第一版补给点包括：出发补给点、居住舱补给柜、氧气生产站、登陆飞船补给舱。
- 不引入 `1` 键，保持原始交互逻辑一致。

体能：

- 体能在火星地表移动时消耗；正常走路时，基础规则约为氧气百分比下降速率的 0.5 倍，并乘以 `1.75` 手感系数，让体能压力比早期版本更明显。
- 持续加速移动时，体能消耗为走路 2 倍，氧气消耗为走路 1.5 倍。
- 火星低重力下推荐跳跃探索，跳跃移动的体能消耗按走路 0.5 倍处理。
- 原地站立时，体能会缓慢恢复，但氧气仍按地表生命维持规则下降。
- 体能低于 20% 时触发生命维持提示，建议回到居住舱休整。
- 进入或离开居住舱时，氧气背包和体能都恢复到 100%。
- 普通氧气补给点只更换氧气背包，不恢复体能。
- 体能归零时，亚历克斯在居住舱内重新同步，氧气和体能恢复满值。

## 8.2 HUD 指标分类

HUD 指标分为两组：

基地：

人员：

## 9. AI 对话方案

建议做成：

固定主线逻辑 + AI 生成自然对白。

也就是说：

- 任务节点、结局、数值变化由游戏代码控制。
- AI 只负责“角色怎么说”。
- AI 不能随意改变主线、发明新设施、提前揭示隐藏剧情。

埃隆是后续开放式 AI 问答的第一位测试角色。当前阶段先使用固定知识库和大型固定对白池，它和普通固定对白角色不同：

- 固定对白仍由项目文件和游戏代码控制，用于剧情节点。
- 每组固定对白有唯一 ID，优先播放未读内容，避免玩家每次聊天都重复。
- 开放问答只回答玩家自由提问，不改变任务、数值、结局和设施状态。
- 第一版返回的 `effects` 保持空数组，避免 AI 回答直接影响玩法。
- 埃隆必须明确：简体中文版它叫“埃隆”，英文版叫 `Elon`；只是同名，不是现实人物本人，也不代表现实人物发言。
- 详细接口和提示词边界见 `docs/ai/elon-agent-design.md`。

### DeepSeek 接入方式

网页前端不能直接调用 DeepSeek，因为 API Key 会暴露。

需要增加一个后端接口：

```txt
前端 Dialogue UI
  -> /api/dialogue
  -> DeepSeek API
  -> 返回角色对白
```

后端保存：

- `DEEPSEEK_API_KEY`
- 模型配置
- 角色卡
- 当前章节知识库

前端只发送：

- 当前角色
- 当前任务节点
- 玩家刚才选择
- 当前基地状态
- 最近几轮对话摘要

### AI 输出限制

AI 返回内容建议统一为 JSON：

```json
{
  "speaker": "史蒂夫",
  "text": "亚历克斯，氧气站压缩机没有破裂，但供电曲线不稳定。先缩小问题，再重启。",
  "choices": [
    { "label": "按史蒂夫的流程检查", "effect": "safe" },
    { "label": "直接手动重启压缩机", "effect": "fast" },
    { "label": "先派维修机器人进入管线区", "effect": "robot" }
  ]
}
```

## 12. 奖励与金币系统

奖励系统分为 `积分` 和 `金币` 两条线：

- 积分用于记录任务完成、建筑探索、隐藏发现和搞笑事件。
- 金币用于未来购物系统，暂时只做随机拾取和累计展示。
- 所有奖励都不提前写进对话或任务提示，只在玩家做到后即时出现反馈。

积分规则：

- 主线任务步骤完成：`+100`。
- 支线任务步骤完成：`+40`。
- 9 个主要建筑首次探索：每个 `+10`，全部探索完成额外 `+100`。
- 带问号的隐藏目标首次发现或对话：`+30`；发现后雷达不再显示 `?`。
- 搞笑事件默认 `+10`，例如用缩放枪放大或缩小太阳。
- 首次把火星足球踢进球门时，除进球积分外一次性获得“相机”。弹出奖励框三行显示：`恭喜！你获得了一台相机`、`随时按 G 键可以使用相机`、`照片会展示在居住舱的墙上`。相机无需建模。
- 获得相机后，按 `G` 打开或收起摄像框；摄像框状态下人物不移动，`A/D` 转向，`W/S` 调整放大倍数，`E/Enter` 拍照。照片保存在居住舱照片墙，最多展示 `7` 张，拍满后新照片替换最旧照片。
- 居住舱内照片墙为实体相框墙，靠近后按 `E` 进入放大查看；查看时 `A/D` 左右切换，`W` 放大，`S` 恢复，`Q` 退出，`E/Enter` 触发当前照片 PNG 下载。浏览器版本不静默保存到桌面。
- 负面搞笑事件默认 `-10`，适合轻松荒诞的错误选择。
- 氧气和体力不足导致的重生暂时不扣积分、不扣金币。

金币规则：

- 同一时间只存在一组随机金币，每组固定 10 枚，并沿地表排成一列。
- 金币随机出现在星球表面，必须避开建筑、飞船、太阳能阵列、石碑、岩石等固体。
- 金币大小接近机器人，只保留金币本体，不加外圈；玩家身体穿过金币即可收取。
- 每枚金币单独播放“叮”音效，并飞向右上角金币数值。
- 金币组 4 小时无人收取会迁移到新位置，不会叠加生成多组。
- 雷达只显示一个金币目标，代表当前金币组中心。

职位晋级规则：

- 亚历克斯初始职位为 `实习巡航员`。
- 完成任意正式任务，或积分达到 `100`，晋级为 `初级宇航员`。
- 后续职位主要由积分决定：`300` 为 `正式宇航员`，`700` 为 `高级宇航员`，`1500` 为 `火星任务专家`，`3000` 为 `火星基地副指挥官`。
- `第一位居民` 是叙事称号，不靠刷分；当前以主线完成且史蒂夫信任、基地状态达到较高值作为解锁条件。
- 职位只升不降；负面搞笑扣分不会让玩家降级。
- 金币不参与职位晋级。

## 13. 多 Agent 分工建议

如果一个人做，建议按阶段顺序推进，不要同时开太多线。

如果多个 agent 协作，适合分成 5 个角色：

### 1. 主策划 / 制作人 Agent

负责：

- 控制项目范围。
- 决定主线任务。
- 审核所有设计是否符合“温和科幻、现实火星基地”的方向。
- 维护本工程文档。

### 2. 剧情与角色 Agent

负责：

- 写角色卡。
- 写章节知识库。
- 写固定剧情节点。
- 控制史蒂夫不变成人类、不变成反派。

### 3. 前端游戏 Agent

负责：

- Three.js 场景交互。
- 任务系统。
- 对话触发。
- UI 状态管理。

### 4. 美术资产 Agent

负责：

- 裁切立绘。
- 统一图片尺寸。
- 生成史蒂夫或其他角色资产。
- 整理 `assets/portraits/`。

### 5. AI 接入 Agent

负责：

- 后端 `/api/dialogue`。
- DeepSeek API。
- prompt 结构。
- JSON 输出校验。
- 网络失败兜底。

小项目里不一定真的需要 5 个人，但用这 5 个职责来拆任务，会比较清楚。
