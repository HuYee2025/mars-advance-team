# 技术方案

## 技术栈

- Vite：轻量开发服务器和构建。
- TypeScript：保证场景代码可维护。
- Three.js：WebGL 渲染、程序化模型、灯光、相机和动画。
- ws：多人 v1 的轻量 WebSocket 同步服务。

## 渲染方案

- 程序化地形：径向网格生成弧面火星地表，顶点色制造沙丘层次。
- 真实太阳光照 MVP：太阳方向改为由真实当前时间计算，不再使用游戏内加速太阳循环。实现入口为 `src/mars-sun-model.ts`，按 NASA Mars24 近似算法计算 Mars Sol Date、MTC、太阳黄经 `Ls`、太阳赤纬、日下点经纬度和火星日心距离；`src/main.ts` 用该日下点方向更新 `DirectionalLight`、可见太阳位置、太阳能阵列朝向和暗面蜘蛛避光判断。太阳直射强度按火星到太阳距离的 `1 / r^2` 变化，并继续叠加沙尘暴衰减；昼夜明暗交给 Three.js 标准材质法线和阴影处理。
- 运行时图片资源：火星地表贴图和对话立绘优先使用 WebP，控制首屏和对话加载体积。正式剧情对话背景使用从 `src/assets/mars-albedo-generated.webp` 裁切压缩出的 `src/assets/dialogue-mars-surface-bg.webp`，作为暗红火星局部俯拍纹理，不直接加载高清原图。
- 程序化建筑：Cylinder、Box、Sphere、Cone 等基础几何组合出基地。
- 动态元素：机器人沿路径巡逻，基地灯光轻微闪烁，粒子表现沙尘。
- 车辆地表特效：火星车使用贴地半透明轮迹和低透明度细雾粒子表现尘土，不做实时地形变形。
- 暗面地貌：星球亮面基地保持平缓；古树和坠毁飞船之间的星球背面局部提高沙丘、浅坑和岩脊起伏，并在蜘蛛巢穴周边增加岩石密度；新增岩石统一使用原有火星红褐岩石材质，明暗只交给环境光照处理，不再制作暗色岩石。车辆 `greatCircleLoop` 路线附近保留平滑安全带，并过滤路线附近随机岩石；远古巨树拱门脚部 collider 继续阻挡玩家/足球，但车辆路线避障忽略这些 collider，保证巡检车和运输车按原路线通畅运行。
- 暗面蜘蛛：使用低多边形程序化八足模型和有限摆腿动画，只作为环境生物自主巡游；不接任务、战斗、碰撞交互或多人同步。蜘蛛活动范围固定在古树和坠毁飞船之间的巢穴周边，不按“暗面”全局扩散；移动时避开建筑、飞船等实体 collider，遇到岩石 collider 会抬高身体和步态表现为爬过；遇到激光剑光源时主动回避并偏回巢穴活动圈。
- 蜘蛛巢穴：第一版采用一个浅的小型地表黑洞作为外部地标，位置固定在远古巨树和坠毁飞船之间的背面区域；通过地形高度函数在巢穴周边做碗形下陷，黑色洞底作为视觉封口，不使用外置圆环，也不真实挖穿球体地形；洞口保留 collider，玩家不能完全进入。雷达未发现前显示绿色 `? / 未知地点`，接近后解锁“暗面蜘蛛巢穴”并奖励 `+100` 积分。
- 轨道星座：`ARES Starlink-M` 使用程序化低多边形卫星和微光点表现火星低轨通信星座，不引入外部模型或真实网络仿真。
- 标签系统：建筑对象投影到屏幕坐标，玩家靠近时显示中文标签。

## 性能目标

- 桌面端接近 60fps。
- 移动端至少 30fps。
- 左侧人员 HUD 的体能数值下方保留实时性能诊断读数，界面名暂包装为“信号延迟 / Signal Lag”，底层按约 `0.5` 秒窗口刷新平均单帧耗时毫秒值；约 `017` 对应 60fps，`033` 对应 30fps，`100` 对应 10fps，即使雷达关闭也保持可见。
- 仅首页底部保留轻量四位数字访客计数器，进入游戏后隐藏。前端访问 `/api/visitors`，由腾讯云服务器上的 Node 服务读取请求 IP，保存 IP hash 到 `data/visitor-stats.json`，同一 IP 多次访问不重复增加累计人数；如果接口未部署，首页底部保持 `0000` 且不影响游戏。
- WebGL 渲染像素比上限为 `1.25`，避免 Retina/大窗口下按过高分辨率持续渲染导致 MacBook Air 发热和帧率大幅波动。
- 相机取景、照片电子屏大图查看、缩放枪瞄准和正式剧情对话属于专注模式；专注模式下不显示普通信息栏、互动弹窗或临时提示框，避免打断当前操作。
- 第一版避免重型后处理、外部高模、复杂物理和实时阴影滥用。
- 后续如果增加模型，应优先使用 glTF/GLB，并控制贴图尺寸。

## 移动端限制

- 小屏幕只保留必要 HUD。
- 虚拟摇杆用于移动，右下角保留较小的圆形“跳”触摸按钮，大按钮左侧贴近放置更小的“加速”触摸按钮，避免遮挡画面；不再提供“照明/头灯”按钮或快捷键；鼠标锁定只在桌面启用。
- 触摸端靠近可交互对象时，直接弹出可点击互动菜单；不要求玩家寻找或模拟 `E` 键。
- 触摸端对话按钮和“继续”按钮必须可直接点击，不能只依赖键盘确认。
- 粒子数量、阴影分辨率和像素比已经做了保守限制。

## 当前 1.0 操作规则

- 移动：`WASD` 或移动端摇杆。
- 首页：标题、说明和入口按钮在宽屏和窄屏都水平居中排列；默认选中“进入基地”。小屏中文主标题保持更大但字重更细，副标题说明和两个入口按钮压小，避免遮挡火星画面。`W/S` 或方向键上下切换按钮；`E` 进入基地；`Q` 直接进入“故事概要”；鼠标悬停只切换当前视觉选中项，不改变键盘快捷键含义。
- 进入互动：桌面端靠近对象后按 `E`；触摸端靠近后点击弹出的互动菜单。
- 互动选择：同一位置最多显示两个最高优先级互动；桌面端用 `Q` 执行左侧、`E` 执行右侧；触摸端普通探索只显示一个最高优先级的明确动作，避免出现“多个可互动项”的综合入口；史蒂夫信息不再显示接收选项，只有时空之门支付这类二选一确认仍显示两个按钮。
- 室内互动：居住舱、温室和飞船内舱属于室内专注空间，不主动弹出外部信息框或外部对象互动。居住舱外必须贴近舱门才显示“进入居住舱”，居住舱主线检查通过进门触发，不再有远距离“检查居住舱”按钮；居住舱内必须靠近白色内部门才显示“离开居住舱”；照片电子屏也只在靠近屏幕时显示查看互动。维修机器人也只在近身距离内显示对话互动，避免和建筑入口叠成综合按钮。
- 物体名称：普通探索状态下不显示世界空间悬浮名称；名称只在互动提示/对话流程和雷达标签中出现，避免场景中长期漂浮文字破坏沉浸感。
- 对话选择：桌面端普通对白用 `E` 推进、`Q` 返回上一句；单个继续按钮不占用 `Q`，仍允许回看上一句；两个并列选项时选择优先，`Q` 选择左侧、`E` 选择右侧。触摸端直接点击对话按钮或“继续”。正式游戏对话最多只允许两个选项，匹配 `Q/E` 二选一操作；需要更多内容时拆成后续节点或改为单个继续按钮，不允许出现第三选项。
- 跳跃：桌面端没有互动选择时按空格；触摸端点击右下角大号“跳”按钮。
- 地图：宽屏/桌面进入游戏后默认打开普通雷达；小屏或触摸端进入游戏后默认关闭雷达。`F` 或右上角圆形 `F` 按钮短按切换普通雷达；长按 `F` 进入临时全屏雷达，背景虚化、游戏世界继续运行，但键盘/人物操作焦点全部交给雷达，人物不可移动也不可触发普通互动；松开 `F` 后恢复到长按前状态。
- 全屏雷达缩放：按住 `F` 的全屏雷达状态下，`Q` 缩小雷达比例以显示更大区域，`E` 放大雷达比例以查看更小区域；放大时圆点和文字轻微变大。全屏雷达状态下 `Q/E` 优先用于雷达缩放，不触发互动选择。
- 小屏地图：点击雷达进入全屏地图。
- 雷达：游戏开始后直接显示已知设施和未知对象；未知生物、未知残骸、黑色方碑和 埃隆 未被发现或对话前统一显示为 `?`，不再需要 `O` 键开启；首次发现或对话后移除 `?`，改用对应分类圆点并显示真实名称。常规分类用纯色小圆点区分，任务目标为黄色圆点并带一圈圈放大渐隐的黄色雷达光圈，氧气补给为绿色圆点，金币为更小的 `$` 金币样式且不发光，太阳显示为黄色发光圆点，避免和任务引导混淆。雷达内不显示 `N/E/S/W` 方位字母；`坐标/Position` 和 `方位/HDG 000°` 朝向读数固定排成一行，显示在右上角金币、积分和 `F` 按钮正下方，但更新逻辑独立于雷达开关，颜色和字号对齐左侧 HUD 的“制氧量”等指标标签。全屏雷达中目标点和标签字更大，便于查看。
- 雷达补氧提示：氧气背包低于 35% 时，只把飞船类氧气源显示为绿色圆点，包括 `01 飞船 登陆飞船`、`02 飞船 货运飞船`、`03 飞船 返回飞船` 和 `坠毁飞船残骸`；居住舱、温室和氧气生产站不再作为外部补氧点显示。靠近飞船实体的“更换氧气背包”互动只在氧气进入 20% 报警区间后出现，未报警时路过不弹按钮。任务目标使用黄色脉冲圆点。
- 体能/氧气：地表行走时氧气按基础速率下降，体能基础消耗在“氧气数值下降速率 0.5 倍”的探索规则上再乘以 `1.75` 手感系数；按住加速移动时，体能消耗为走路 2 倍，氧气消耗为走路 1.5 倍；跳跃探索按走路 0.5 倍消耗体能，作为火星低重力下的推荐探索方式；原地站立会缓慢恢复体能但仍消耗氧气；进入或离开居住舱、进入温室时氧气和体能都恢复到 100%。成功搭乘巡检车或运输车会立即补满氧气和体能，乘车期间不消耗氧气和体能。
- 头盔灯：已从当前版本移除，不再提供“照明”按钮、头灯快捷键或多人同步状态。
- 视角：按 `C` 在第一人称和默认第三人称之间切换；使用 `=` 和 `-` 调整镜头远近后，按 `0` 一键恢复默认第三人称视角。
- 背景音乐：进入游戏后默认播放；按 `B` 可切换播放/停止。虫洞坠落期间强制静音，结束后仍遵守玩家当前的 `B` 键开关状态。
- 首页镜头：首页展示火星时使用 `TITLE_PLANET_VIEW_SCALE = 0.7`，让标题页背景星球比游戏内实际星球视觉尺寸小 30%，减少首屏压迫感；游戏内星球半径和布局不受影响。
- ARES 火星历：首页火星日期不再是静态文本。采用 NASA Mars24 的平均火星太阳日 `88775.244` 秒作为一个 sol；以网站上线显示日 `2026-06-27 00:00 Asia/Shanghai` 对应 `2050 年 / 第 30 火星日` 为 ARES 基地民用历锚点。之后每经过一个真实 sol，火星日 +1；每 `669` 个 sol 进位一个 ARES 火星年。该规则参考 NASA GISS Mars24 算法说明和 NASA/JPL 火星任务资料中 “sol = 24h 39m 35.244s / 火星年约 687 地球日、约 668.6 sols” 的基准；游戏内显示采用整数 sol，故民用年按 669 sols 取整。
- 太阳光照时间：太阳光照系统使用真实 `Date.now()`，网页运行时按真实秒推进，不再按 `10x` 游戏时间加速。为了符合这个规则，沙尘暴循环也从原先 `elapsedTime * 10` 改为真实 `elapsedTime` 秒推进。
- 星球尺度：火星视觉半径使用 `WORLD_EXPANSION = 1.5` 放大 50%，布局坐标同步展开 50%；建筑、飞船、机器人、车辆、足球等模型尺寸和交互/碰撞半径不随星球放大，避免基地显得过度拥挤。
- 远古巨树拱门虫洞：时空之门不再定时自动打开；金币不足 `50` 时靠近门洞不弹支付信息。金币足够时，玩家进入门洞约 `1.2x` 判定区会弹出“你是否愿意为穿越时空之门支付50个火星币？”，飞行背包状态下处在门洞投影上方同样可触发该支付邀请，选项为 `考虑一下(Q)` / `支付(E)`；选择“考虑一下”后必须离开判定区再进入才会重新弹出，选择“支付”后扣 `50` 金币并开启约 `28.5s`。支付只开启门，不直接进入虫洞；玩家步行或飞行穿过发光门的中轴核心区后，才进入约 `2s` 的白色粒子云过渡，靠近门洞边缘不触发：密集白色粒子形成近似白光，屏幕偏下居中显示黑色粗体字幕“这是……”，随后粒子向四周和深处快速散开并露出黑洞，再触发约 `30s` 虫洞坠落。虫洞期间暂停背景音乐并保持寂静，隐藏普通世界、金币实体、雷达金币标记和所有 2D HUD/雷达/任务/互动/标签界面，只保留虫洞 WebGL 画面与人物；第三人称镜头默认让亚历克斯比上一版上移约 `10%`，人物尺寸比上一版虫洞镜头缩小约 `70%`；虫洞中 `A/D` 和方向键左右控制横向漂移，`W/S` 改为前后透视控制。`W/S` 不是立即突进/突退，而是通过深度速度、阻尼和边界阻力逐步推进：按住 `W` 时人物向洞内平缓前进并缩小到默认虫洞尺寸的约 `70%`；按住 `S` 时人物相对平缓后退并放大到默认虫洞尺寸的约 `130%`。黑洞几何体变化、粒子流动和旋转保持默认速度，不再受 `W/S` 前进或后退影响。横向漂移也不是绝对直线，控制中心会叠加轻微周期性波动和扰动力，形成有阻力的不稳定漂浮感；虫洞期间 `E` 暂停黑洞下落进度，`Q` 继续下落。虫洞前约 `20s` 使用算法生成的有机循环隧道，默认视觉前进、粒子流动和旋转速度整体为上一版约 `2x`，后约 `10s` 衔接红色火星俯视地表快速放大的坠落段，最终无伤安全落回出生点；每次完成穿越奖励 `+200` 积分。
- ARES Starlink-M：当前取消按火星日批次释放任务，不再每 `24` 个火星日新增卫星；场景初始化时直接固定生成 `5` 组星链，每组 `7` 颗，共 `35` 颗，分别在不同轨道倾角和升交点经度上循环飞过火星上空。每组保持同轨道面和小相位间距，视觉上接近从地表看到的一串星点。为符合当前画面可读性，星链不再按真实低轨比例贴近火星表面，而是使用高于近火流星轨道的游戏可视高度，基础高度约为高出地表 `190` 游戏单位；卫星实体代理放大约 `5` 倍，材质改为白色自发光，并叠加更亮的圆形微光点和点光源。雷达/任务 HUD 显示轨道链路状态，不做释放动画。每颗星链卫星都是缩放枪可命中的实体目标，首次命中奖励隐藏发现积分。
- 基地布局：主要建筑和三艘飞船集中在首页亮面，形成“居住舱中心 + 工业/仓储/飞船环绕”的基地组团；三组太阳能电池阵列平行排成一条赤道式电力场，放在首页可见、靠近 03 返回飞船一侧的亮面受光区域，约位于源码坐标 `(-22,52) / (-4,52) / (14,52)`，并远离温室、医疗舱和飞船；坠毁飞船残骸、福福救援点、黑色方碑、足球、NASA 机遇号火星车遗迹和其他小型/神秘物体不纳入主基地搬迁。

- 积分系统：前端运行时维护 `scorePoints`、`awardedEvents` 和建筑/隐藏发现记录。主线任务默认 `+100`，支线任务默认 `+40`，建筑探索默认 `+10`，隐藏问号目标默认 `+30`，机遇号火星车遗迹首次发现为 `+50`，搞笑事件默认 `+10`，负面搞笑事件默认 `-10`。建筑探索积分到达范围后自动发放，不显示确认按钮或浮动提示；如果玩家首次撞到建筑、飞船或太阳能阵列等实体碰撞体，也会立即触发对应的一次性探索积分，避免交互点半径和模型阻挡范围不一致导致漏判。神秘物体包括足球、NASA 机遇号火星车遗迹、黑色方碑、未知生命和坠毁/未编号异常物体；未发现前在雷达上用 `?` 标记并只显示“未知物体/未知生命迹象”，首次接近、碰撞或对话发现后奖励隐藏发现积分，随后雷达移除 `?` 并显示真实名称。后续新增任务应优先走统一奖励 API，只有特殊事件才单独覆盖分值。
- 职位系统：人员 HUD 上方显示动态职位，初始为 `巡航员`；下方显示玩家名，中文模式为 `亚历克斯`，英文模式为 `Alex`。前端运行时维护 `currentRank`、`completedAnyTask` 和 `rankRules`；完成任意正式任务或积分达到 `100` 后晋级 `初级宇航员`，后续按 `300 / 700 / 1500 / 3000` 积分晋级。职位只升不降，金币不参与晋级；最高称号 `第一位居民` 暂由主线最佳结局条件解锁。
- 史蒂夫呼叫：收到基地 AI 信息时不自动弹出信息栏，也不显示“是否接收”的选项；只让顶部 `R` 按钮进入未读红光提示。玩家按 `R` 后直接打开史蒂夫对话框；在此之前提示保持待处理状态，不自动收起或重发。
- 随机金币：前端运行时维护 `3` 个金币组，每组 10 枚并沿地表切线排成一列，刷新周期为真实游玩时间 `2` 分钟。生成时在星球表面随机抽样，并根据 `world.colliders` 避开建筑、飞船、太阳能阵列、石碑、岩石、古树拱门等固体，也避免和足球、福福及其他金币组重叠。金币模型正反两面都有放大的 `$` 标记，雷达显示每组金币中心的 `$` 标记。
- 福福：首次靠近残骸保温层旁的福福后按 `E` 安抚，福福进入已救援状态并自动跟随亚历克斯；福福低多边形模型已比上一版缩小 `50%`，贴地高度和安抚互动范围同步收窄；找到福福时奖励 `+40` 积分，并把后续宇航服氧气消耗速度降为原来的 `75%`；取消按 `Q` 停留/脱离逻辑，不论玩家跑多远、跑多久，福福最终都会继续追上玩家。
- 火星足球：居住舱外放置一个可碰撞足球。足球使用程序化白色足球贴图、黑色拼块、缝线和 bump 细节；运动采用轻量球面物理，竖直方向使用火星重力 `3.71 m/s²`，地表方向按小星球切线滚动。亚历克斯、机器人、火星车和运输车会按接触速度给足球冲量；足球的踢出冲量和最高速度按偏夸张的游戏手感处理，约为早期版本的 2 倍。建筑、飞船、岩石等 `world.colliders` 固体会让足球反弹并损失部分能量。靠近静止或运动中的足球时可按 `E` 拾取，拾取后足球固定在玩家身前并暂停球体物理，按 `Q` 放回地面恢复碰撞和进球判定。足球出生点穿过星球直径的另一面放置球门，落点会避开障碍物；足球进入球网后每次奖励 `+100` 积分，并立即回到出生点。首次进球还会一次性解锁相机奖励。

## 部署方案

当前 1.0 的单人体验仍是纯前端静态网页游戏，构建后产物在 `dist/`。多人 v1 额外需要一个常驻 Node WebSocket 服务，用于同步公共大厅中其他玩家的位置和外观状态；该服务不保存任务进度，不需要数据库。
公开故事页图片位于 `public/story-assets/`，线上部署时使用 WebP 版本，避免复制未引用的大 PNG 副本。

推荐部署方式：

- 本地执行 `pnpm build` 或 Codex 内置 pnpm 构建。
- 上传 `dist/` 内容到服务器静态站点目录。
- 使用 Nginx 托管根路径。
- 如启用多人模式，运行 `pnpm dev:server` 或线上 Node 进程启动 `server/multiplayer.ts`，并让 Nginx 将 `/ws` 反向代理到该进程。

详细步骤见：`docs/deployment.md`

## 多人同步方案

多人 v1 采用“客户端权威位置广播 + 服务端房间转发”：

- 公共房间固定为 `public-mars-alpha`。
- 前端身份存储：
  - `localStorage["mars.playerId"]`
  - `localStorage["mars.displayName"]`
- 前端通过 `src/multiplayer.ts` 连接 WebSocket：
  - 开发环境默认：`ws://127.0.0.1:8787/ws`
  - 线上默认：同域名 `/ws`
  - 可用 `VITE_MULTIPLAYER_WS_URL` 覆盖。
- 客户端约每秒 12 次发送本地玩家快照：位置、四元数、速度、飞行状态和室内/室外状态。
- 服务端 `server/multiplayer.ts` 只维护在线玩家列表、心跳超时和状态广播；不判定任务、不写数据库、不保存长期进度。
- 左侧人员 HUD 通过多人客户端的远程玩家列表显示在线人数，最低显示 `1人`。
- 远程玩家复用 `createMarsEngineer()` 生成宇航员模型，并在本地做插值平滑。
- 远程玩家不可碰撞、不可互动，不进入任务系统。
- 当 WebSocket 服务不可用或断开时，前端安静重试，单人游戏继续运行。

## 音频方案

背景音乐使用原 MP3 的压缩版：

- 源文件：`星际低语_no-watermark (1).mp3`，约 10MB，只保留在本地，不提交 GitHub。
- 项目内文件：`src/assets/audio/mars-background-light.mp3`，64kbps，约 2.5MB。

原因：

- 保留原背景音乐的听感。
- 减小部署包体积。
- 保持纯前端静态部署，不增加后端依赖。
