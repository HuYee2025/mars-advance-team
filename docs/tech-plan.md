# 技术方案

## 技术栈

- Vite：轻量开发服务器和构建。
- TypeScript：保证场景代码可维护。
- Three.js：WebGL 渲染、程序化模型、灯光、相机和动画。
- ws：多人 v1 的轻量 WebSocket 同步服务。

## 渲染方案

- 程序化地形：径向网格生成弧面火星地表，顶点色制造沙丘层次。为优先保证网页加载速度和游戏流畅度，火星球体地形网格暂保持 `96 x 54` 分段，只关闭地表材质的 `flatShading`，用平滑顶点法线减少近距离三角面感。
- 真实太阳光照 MVP：太阳方向改为由真实当前时间计算，不再使用游戏内加速太阳循环。实现入口为 `src/mars-sun-model.ts`，按 NASA Mars24 近似算法计算 Mars Sol Date、MTC、太阳黄经 `Ls`、太阳赤纬、日下点经纬度和火星日心距离；`src/main.ts` 用该日下点方向更新 `DirectionalLight`、可见太阳位置、太阳能阵列朝向和暗面蜘蛛避光判断。太阳直射强度按火星到太阳距离的 `1 / r^2` 变化，并继续叠加沙尘暴衰减；昼夜明暗交给 Three.js 标准材质法线和阴影处理。
- 大型物体阴影：取消此前的半透明椭圆假投影层，改回 Three.js 真实 shadow map。`src/main.ts` 的太阳 `DirectionalLight` 阴影相机跟随玩家附近区域，扩大 `near/far` 和正交覆盖范围，让玩家附近的飞船、基地建筑、远古巨树拱门、黑色方碑、NASA 火星车以及近火巨型流星主体都能尝试投射真实阴影。远处两颗小流星距离太远，不纳入常驻阴影范围。该方案仍要控制 shadow map 尺寸，优先保证可见近景空间感和帧率。
- 自适应画质：内部按约 `0.5s` 窗口统计平均单帧耗时，桌面默认高画质，触摸设备默认均衡；画质档同时控制像素比、阴影范围、动态灯数量、LOD 可见距离和 UI 刷新频率。左侧人员 HUD 保留“信号延迟”的火星任务叙事，但直接以该采样值换算本机 `FPS`，不再把多人网络延迟混进性能读数。
- 运行时图片资源：火星地表贴图和对话立绘优先使用 WebP，控制首屏和对话加载体积。正式剧情对话背景使用从 `src/assets/mars-albedo-generated.webp` 裁切压缩出的 `src/assets/dialogue-mars-surface-bg.webp`，作为暗红火星局部俯拍纹理，不直接加载高清原图。
- 核心 GLB LOD：`public/models/core/` 保存亚历克斯、维修机器人、车辆和居住舱的实际 GLB；每个文件包含 `LOD0/LOD1/LOD2`，最多 3 个共享材质，并由 `src/core/glb-lod-system.ts` 按模型距离切换。GLB 在主世界加载后异步请求，加载失败保留程序化 fallback；通过 `npm run assets:models` 可重建二进制资产。
- 静态合批：`src/core/static-batching.ts` 按静态根节点、材质、阴影和 renderOrder 合并基地几何；门、升降梯、太阳能追踪板、虫洞和室内节点必须排除。当前 504 个静态来源合成 76 个批次。星链卫星单独使用 InstancedMesh/Points，不再为每颗卫星创建五个可见 Mesh 和独立动态灯。
- 程序化建筑：Cylinder、Box、Sphere、Cone 等基础几何组合出基地。
- 动态元素：机器人沿路径巡逻，基地灯光轻微闪烁，粒子表现沙尘。
- 维修机器人避障：维修机器人巡逻点生成时必须远离建筑/飞船/阵列等固定 collider；运行时在初始化、等待和移动三个状态都要重新执行固定 collider 推离修正，避免机器人贴墙后穿进建筑体。机器人专用安全距离应大于玩家半径，优先保证视觉上不进入建筑模型。
- 车辆地表特效：火星车使用贴地半透明轮迹和低透明度细雾粒子表现尘土，不做实时地形变形。
- 暗面地貌：星球亮面基地保持平缓；古树巨洞附近的蜘蛛活动区不再使用贴地熔岩/粒子云 shader，因为近距离会显得像平面贴图。后续如果重做裂缝，必须用真实窄沟几何、局部地形下陷或可见岩壁边缘，而不是单张发光平面。蜘蛛活动区附近岩石密度降低并避开中心区，减少蜘蛛被岩石和地形夹住的概率；新增岩石统一使用原有火星红褐岩石材质，明暗只交给环境光照处理，不再制作暗色岩石。车辆 `greatCircleLoop` 路线附近保留平滑安全带，并过滤路线附近随机岩石；远古巨树拱门脚部 collider 继续阻挡玩家/足球，但车辆路线避障忽略这些 collider，保证巡检车和运输车按原路线通畅运行。
- 暗面蜘蛛：使用低多边形程序化八足模型和有限摆腿动画，平时作为环境生物自主巡游，不接任务或多人同步。蜘蛛从新开局起始终可见，不受主线或异常区门控；动态 collider 同样始终启用，接触伤害还必须检查 `spider.group.visible`，防止未来任何临时隐藏状态产生隐形攻击。蜘蛛设定为通过远古巨树拱门时空之门来到火星，没有巢穴；活动范围围绕远古巨树拱门，运行时根据真实太阳方向自动选择古树背光一侧活动，不按固定坐标长期停留在亮面。移动时避开建筑、飞船等实体 collider，遇到岩石 collider 会抬高身体和步态表现为爬过；遇到太阳强光时向古树暗面侧迁移，遇到激光剑光源时主动回避并偏回活动区。每只蜘蛛都要保留独立动态 collider 和个人巡游偏移，彼此之间持续执行分离推力，避免多只蜘蛛重叠成一个模型或集体同向移动。红眼蜘蛛可追踪地面宇航员；触碰按 `20 * damageFactor` 扣体能，体能归零从居住舱重生。飞行状态免疫；激光剑可用且展开时蜘蛛回避并不造成伤害；缩放枪命中红眼蜘蛛后同时改变模型缩放和 `damageFactor`，倍率过期或清除后恢复。
- 碰撞体调试规则：远古巨树拱门 collider 必须同时使用本地 `localX/localZ` 投射到球面，避免左右实体 collider 叠到中轴形成空气墙；只允许覆盖左右门柱和可见低矮根脚的地面 footprint，不能把上方拱形轮廓投影到地面，也不能在门洞下方或实体外侧放额外根部 collider。坠毁飞船残骸位于暗面时，collider 必须从 `CRASHED_SHIP_SITE_NORMAL` 沿局部切线投射，再转换为当前圆形碰撞系统使用的 `center`，不能重新用正面 `planetNormal(x,z)` 推导，否则会把残骸碰撞体投到错误半球，表现为残骸可穿过、另一侧出现透明空气墙。残骸 collider 只覆盖可见船体核心，不做过度外扩，避免玩家在残骸前方空地被透明障碍挡住。
- 蜘蛛雷达：不再生成“暗面蜘蛛巢穴”雷达未知点，也不再奖励对应隐藏发现积分。
- 轨道星座：`ARES Starlink-M` 使用程序化低多边形卫星和微光点表现火星低轨通信星座，不引入外部模型或真实网络仿真。
- 首页与轨道模式共用远景星点天幕：`6200` 个、非距离衰减点径 `2.05`、不透明度 `1`，分布在火星外 `1400–2300` 的远景壳层；星点材质不受地表沙尘雾影响。这样 Safari 高分屏首页可读，同时战机不会穿进星点形成近景颗粒。
- Shader 特效借鉴边界：阅读 `Three 学习资料/一个 Shader 生成整个世界：Three.pdf` 后，确认可借鉴 `Ray Marching`、`SDF`、`fBm`、shader 内粒子、体积辉光近似、距离雾和 film grain，但只用于特殊视觉层，不替换主世界实体渲染。虫洞坠落曾尝试叠加全屏 shader veil，沙尘暴曾尝试新增 shader 球幕风带；两者近看都容易暴露贴图/屏幕层痕迹，已从 `src/main.ts` 回退。当前虫洞坠落保留原有粒子、黑洞、有机隧道、火星坠落和旧 veil 贴图，人物操作不变；沙尘暴保留原有雾、背景色变化、旧沙尘粒子和 `stormStrength` 逻辑，不再使用新增球幕风带。远古巨树拱门时空之门的能量膜仍使用流动 shader，并增加门内深度 veil，保留原有粒子、电弧和光源；远处太阳已从多层几何球壳改为发光 sprite 光斑，远景显示接近纯白的过曝光芯、外发光和淡光芒；缩放枪放大太阳到 `2x` 和 `3x` 时仍只放大默认发光太阳，不叠加近景贴图，到接近 `4x` 时才用默认太阳白色光盘作底，并在上方覆盖 `src/assets/sun-close-detail-texture.png` 透明近景太阳图。该贴图使用用户提供的圆形透明太阳主体图，只保留太阳圆盘和贴身亮边，不带周边日冕喷溅；底层白盘复用默认太阳光盘纹理，尺寸略大于贴图，用于填补边缘漏光。太阳 sprite 使用高 `renderOrder`，但必须保留 `depthTest`，让飞船、建筑、星球地平线等真实模型可以遮挡太阳；避免太阳穿透飞船。此前尝试的 canvas 动态火纹 sprite 因产生灰点和额外纹理更新开销已移除。近距图显现时远处小太阳层同步退场。近距太阳不再叠加远景射线、大面积朦胧光晕、均匀圆环、额外遮罩或会压暗主体的低透明纹理层。太阳缩放倍率同时影响 `DirectionalLight` 强度、真实 shadow map 光源位置和全屏过曝层：`2x` 开始明显增强，接近 `4x` 时模拟火星表面亮到难以直视；太阳达到 `3x` 以上时，体能消耗按太阳当前倍率加速，即使处于对话、室内、乘车或电梯等原本暂停体能的状态也生效，太阳恢复默认大小后取消该热耗。太阳继续作为雷达和缩放枪可锁定目标。蜘蛛核心活动区曾尝试贴地熔岩 shader，但因近距离像平面贴图已取消。凡是需要碰撞、雷达、缩放枪命中、任务触发、照片截图或多人同步的对象，仍必须保留 Three.js 实体或明确数据对象。详细研究笔记见 `docs/research/shader-world-threejs-procedural-scene.md`。
- 标签系统：建筑对象投影到屏幕坐标，玩家靠近时显示中文标签。

## 性能目标

- 蜘蛛过程式步态收口：参考 `majidmanzarpour/threejs-procedural-spider` 的“脚端决定身体高度、移动和步态共享实际位移、腹部通过腰节独立摆动”原则，但不移植完整逐腿 IK、射线落脚搜索或物理系统。蜘蛛根节点保持低离地高度，取消额外上下 bob；威胁状态只依据宇航员是否实际贴地，不依据飞行开关。贴地时在约 `68` 单位内取得目标、约 `96` 单位内持续追击，最近者直追、其他侧翼包抄，并在攻击态解除巡逻领地约束和使用更小分离距离；离地后眼点恢复绿色，蜘蛛仅按各自背光巡逻逻辑移动。接触玩家时触发红色屏幕边缘闪烁、沿球面击退约 `5.4` 单位和短全局伤害冷却。

- 桌面端接近 60fps。
- 移动端目标不低于 45fps；低于阈值时优先降低阴影、LOD 可见距离和动态灯，再降低像素比。
- 左侧人员 HUD 的“信号延迟 / Signal Lag”用本机实时渲染 `FPS` 作为飞行器/宇航服端的链路稳定度读数；多人 WebSocket 是否连接不影响此数值。网络 `ping/pong` 往返时间不再占用该位置，帧耗、draw calls、三角形和碰撞体数量仍可通过开发调试接口查看。
- 仅首页底部保留轻量四位数字访客计数器，进入游戏后隐藏。前端访问 `/api/visitors`，由腾讯云服务器上的 Node 服务读取请求 IP，保存 IP hash 到 `data/visitor-stats.json`，同一 IP 多次访问不重复增加累计人数；如果接口未部署，首页底部保持 `0000` 且不影响游戏。
- WebGL 渲染像素比默认上限为 `1.25`，并由自适应画质在帧耗持续偏高时下调；阴影图同步由 `2048` 降档，避免 Retina/大窗口下按过高分辨率持续渲染导致 MacBook Air 发热和帧率大幅波动。
- 普通 HUD、标签、坐标和雷达标记不再逐帧操作 DOM，而是按当前画质档每约 `100-166ms` 刷新；雷达打开时仍实时反映玩家移动，但不再每秒重建数十次标记节点。
- 相机取景、照片电子屏大图查看、缩放枪瞄准和正式剧情对话属于专注模式；专注模式下不显示普通信息栏、互动弹窗或临时提示框，避免打断当前操作。
- 第一版避免重型后处理、外部高模、复杂物理和实时阴影滥用。
- 高步数 ray marching 不作为移动端常驻渲染方案。若用于虫洞、传送门或特殊全屏镜头，必须控制步数、分辨率、持续时间和像素比，并在移动端保留降级参数。
- 后续如果增加模型，应优先使用 glTF/GLB，并控制贴图尺寸。

## 移动端限制

- 小屏幕只保留必要 HUD。
- 虚拟摇杆用于移动，右下角保留较小的圆形“跳”触摸按钮，大按钮左侧贴近放置更小的“加速”触摸按钮，避免遮挡画面；不再提供“照明/头灯”按钮或快捷键；鼠标锁定只在桌面启用。
- 触摸端靠近可交互对象时，直接弹出可点击互动菜单；不要求玩家寻找或模拟 `E` 键。
- 触摸端对话按钮和“继续”按钮必须可直接点击，不能只依赖键盘确认。
- 粒子数量、阴影分辨率和像素比已经做了保守限制。

## 当前 1.0 操作规则

### 轨道防御专注模式

- `src/orbital-defense.ts` 独立维护 `idle / launch / combat / success / failed / return` 状态、战机姿态、外环陨石、撞击目标、激光束和爆炸。
- `orbitalDefense.unlocked` 默认是 `false`；只有 `finishWormholeFall()` 首次完成星际之门穿越时调用 `setUnlocked(true)`。未解锁时停机模型隐藏且不参与互动扫描。
- X 翼程序化模型由独立结构件组成：六边形锥形机鼻、中央机身、座舱罩/框、宇航技工机器人、四个带进气环与尾喷口的引擎、四片 S 翼、四门翼尖炮和汇聚尾焰。地面实例共三架，灰白机体分别使用红、蓝、黄涂装；`sFoilPivots` 保存每片机翼与对应尾焰 pivot 的战斗展开角，停放时 deployment 为 `0` 且 `engineGlow.visible = false`。玩家选择的停放实例只把涂装色同步到同一个轨道战机和镜头，不复制三套飞行状态机。
- 轨道任务复用现有 Three.js `scene` 与火星球体，不建立第二套 renderer 或第二个关卡页面。
- 进入时保留 `ARES Base Alpha` 供完整起飞镜头使用：`launch` 阶段约 `7.2s`，前 `1.5s` 沿地表滑跑，随后位置同时沿大圆弧增加角度和高度，形成贴着火星曲率向离地 `500` 的环形轨道爬升的连续路径。全程复用同一台第三人称追尾相机和同一架战机，不插入远景、黑场或第二套巡航镜头；任务开始时三架停放模型全部隐藏，返航后恢复，避免另外两架被高空镜头放大拍入画面。
- 到达轨道后默认沿环形轨道持续巡航：位置保持离地 `500`；鼠标控制转向，`A/D` 控制左右飞行，`W/S` 控制垂直上/下飞行，`Shift / Ctrl` 分别临时切至 `76 / 16 u/s`，松开平滑回到 `34 u/s`。按住移动键时不强制投影回轨道，允许真实改变位置；松开后才按每秒最多 `24` 单位的限速方式回归，避免远距离修正带来跳动。战机、相机组成屏幕稳定的飞行坐标系：机身航向始终投影在屏幕水平面；`A/D` 仅额外带最多 `14°` 的短暂侧身反馈，上下移动绝不以火星法线或纵向输入校正机身。碰撞爆炸保留约 `0.9s` 后直接结束任务并回到重生点，不走返航路径。
- `Q` 在 `combat / patrol` 任意阶段可从当前位置自动导航到停机位；轨迹按距离连续靠近和下降，但维持当前追尾镜头、战机姿态与机翼状态，不启动独立返航动画。返回前会隐藏任务目标；未完成任务时是主动中止，不结算成功。
- 飞行、加速和常规光枪没有能量消耗。战斗开始启动所有危险陨石共用的 `115s` 全局倒计时；归零失败，全部击毁后停止计时进入无限巡航。HUD 用 `TIME` 取代 `ENERGY`，剩余 `20s` 进入红色警戒。

- 随机金币：前端运行时维护 `3` 个金币组，每组 10 枚并沿地表切线排成一列，刷新周期为真实游玩时间 `2` 分钟。生成时在星球表面随机抽样，并根据 `world.colliders` 避开建筑、飞船、太阳能阵列、石碑、岩石、古树拱门等固体，也避免和足球、福福及其他金币组重叠。金币模型正反两面都有放大的 `$` 标记，雷达显示每组金币中心的 `$` 标记。

- 火星足球：居住舱外放置一个可碰撞足球。足球使用程序化白色足球贴图、黑色拼块、缝线和 bump 细节；运动采用轻量球面物理。时空门可被发现，但需支付 `50` 金币后限时开启；足球穿过门洞每次奖励 `+100` 经验值并回到出生点。

## 部署方案

当前 1.0 的单人体验仍是纯前端静态网页游戏，构建后产物在 `dist/`。多人 v1 额外需要一个常驻 Node WebSocket 服务，用于同步公共大厅中其他玩家的位置和外观状态；该服务不保存任务进度，不需要数据库。
公开故事页图片位于 `public/story-assets/`，线上部署时使用 WebP 版本，避免复制未引用的大 PNG 副本。

推荐部署方式：

- 本地执行 `pnpm build` 或 Codex 内置 pnpm 构建。
- 上传 `dist/` 内容到服务器静态站点目录。
- 使用 Nginx 托管根路径。
- 如启用多人模式，运行 `pnpm dev:server` 或线上 Node 进程启动 `server/multiplayer.ts`，并让 Nginx 将 `/ws` 反向代理到该进程。

详细步骤见：`docs/deployment.md`

## 多人同步方案

多人 v1 采用“客户端权威位置广播 + 服务端房间转发”：

- 公共房间固定为 `public-mars-alpha`。
- 前端身份存储：
  - `localStorage["mars.playerId"]`
  - `localStorage["mars.displayName"]`
- 前端通过 `src/multiplayer.ts` 连接 WebSocket：
  - 开发环境默认：`ws://127.0.0.1:8787/ws`
  - 线上默认：同域名 `/ws`
  - 可用 `VITE_MULTIPLAYER_WS_URL` 覆盖。
- 客户端约每秒 12 次发送本地玩家快照：位置、四元数、速度、飞行状态和室内/室外状态。
- 服务端 `server/multiplayer.ts` 只维护在线玩家列表、心跳超时和状态广播；不判定任务、不写数据库、不保存长期进度。
- 左侧人员 HUD 通过多人客户端的远程玩家列表显示在线人数，最低显示 `1人`。
- 远程玩家复用 `createMarsEngineer()` 生成宇航员模型，并在本地做插值平滑。
- 远程玩家不可碰撞、不可互动，不进入任务系统。
- 当 WebSocket 服务不可用或断开时，前端安静重试，单人游戏继续运行。

## 音频方案

背景音乐使用原 MP3 的压缩版：

- 源文件：`星际低语_no-watermark (1).mp3`，约 10MB，只保留在本地，不提交 GitHub。
- 项目内文件：`src/assets/audio/mars-background-light.mp3`，64kbps，约 2.5MB。
- 新音乐与音效的创作提示词统一维护在 `docs/audio-prompts.md`。长背景音乐优先用外部 AI 音乐工具生成，短音效可用外部音效生成工具或本地 Web Audio / ffmpeg 二次制作。
- 外部生成提示词不得直接引用现成电影、作曲家或 IP 名称；激光剑等科幻音效只能借鉴声音构成逻辑，不直接采样电影原声。

原因：

- 保留原背景音乐的听感。
- 减小部署包体积。
- 保持纯前端静态部署，不增加后端依赖。

## AT-AT 步行载具

- 2026-07-11 试玩开关：默认让全部可玩设备与异常内容可发现、可触发（飞行背包、缩放枪、激光剑、相机、X 翼和时空门），但不跳过使用门槛；时空门仍需支付 `50` 金币后限时开启，AT-AT 仍需 `500` 经验并支付 `100` 金币。不改变主线任务完成顺序；右侧埃隆对话立绘水平镜像以保持双方对视；AT-AT 黑色脚腕关节半径放大 50%，与脚掌装甲重叠连接。

- `src/world/at-at.ts` 负责 22.5 米程序化模型、四腿关节、步态、落脚沙尘和驾驶资格纯函数；`MarsWorld.atAt` 暴露模型根节点、驾驶舱/登车/下车锚点、足部碰撞锚点、球面方向和移动状态。
- 主循环只在驾驶状态读取移动输入：前进 `4.2m/s`、快速行进 `7m/s`、倒车 `2.4m/s`，使用缓慢加减速和低转向角速度；每帧重新投影到火星球面法线，并以 `7.2m` 主占地半径阻止穿过大型 collider。
- 四个足部使用独立动态 collider，宇航员可走到登车点但不能穿过脚掌；AT-AT 自身移动过滤普通红色岩石和暗面蜘蛛，保留建筑、飞船及巨型结构阻挡。
- AT-AT 第一人称镜头下压约 `0.16rad` 并隐藏红色前视窗遮挡；退出提示最多显示 `5s`。`J / Space` 让左右炮管交替发射蓝色激光，炮管保留低强度红色轮廓光并在驾驶舱短暂补蓝光；转向只改变机体朝向，不驱动炮塔。
- 行走步态提高膝部弯曲和身体晃动，足部落地生成大脚印与短时尘土；脚印使用透明低模贴花池，避免额外高模开销。
- 驾驶 AT-AT 时 `X` 不再打开缩放枪；主炮使用从炮口射出的短段蓝色激光弹，以约 `42m/s` 向前飞行并淡出，长按 `J / Space` 按 `0.28s` 冷却间隔左右炮交替连发。炮管发射后沿机体后方产生 `0.34m` 后坐位移，再平滑回到基准位置。
- AT-AT 足部与蜘蛛碰撞时直接触发踩踏击杀；蜘蛛保留 `3.2s` 的肚皮朝上、四脚朝天、灰色眼睛死亡状态后再隐藏，并立即禁用其碰撞体。
- 驾驶资格统一读取现有 `scorePoints` 与 `coins`：先检查 `500` 积分，再检查 `100` 金币；成功登机后同步扣费，驾驶本身不写奖励事件。开发环境可用 `?atat-preview` 直接到登车点并准备测试额度。
