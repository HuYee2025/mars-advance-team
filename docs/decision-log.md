### 线上图片加载速度优化

原因：

用户反馈互联网页面打开时图片加载偏慢，重点包括火星星球贴图、故事概要页场景概念图和角色贴图。排查发现多张 PNG 在 1.4MB 到 2.8MB 之间，线上静态包会直接下载这些大图。

决策：

- 游戏运行时贴图和对话立绘切换为 WebP：火星星球贴图、Alex、Mother、维修机器人、Elon 对话图均改为 WebP 引用。
- 故事概要公开资源 `public/story-assets/` 全部改为 WebP，并删除未引用 PNG 副本，避免部署包继续携带大文件。
- 故事源页面 `docs/story/core-story-v2-visual.html` 同步改用 WebP 资源。
- 项目内概念图也补齐 WebP 版本，文档默认引用压缩图，避免继续把 1MB 以上 PNG 当作预览路径。
- 压缩策略：在尽量保持清晰度的前提下，按实际展示尺寸适度降采样；透明角色图保留 alpha 通道。
- 构建结果中，大图从约 1.4-2.8MB 降到约 55-256KB，火星贴图约 189KB；概念图压缩后约 83-341KB。

### 黑色石碑赠送放大缩小枪概念

原因：

用户希望把远端黑色石碑从单纯异常点扩展为可互动彩蛋：玩家在石碑处对话后获得一把“放大缩小枪”，后续可瞄准物体并放大或缩小。

决策：

- 当前场景已存在黑色石碑，位置记录在 `src/world.ts` 的 `world.monolith`，此前仅作为远端异常点和声音触发点。
- 黑色石碑改名为“黑色方碑”，形状改成更宽的直立矩形，去掉底座，仅保留暗色接触阴影。
- 方碑位置从 `(260, -315)` 移到 `(-360, -300)`；按现有建筑/飞船/太阳能阵列表面弧长计算，最近距离从约 `19` 提升到约 `32`，不再靠近医疗舱。
- 放大缩小枪第一阶段先作为概念资产入库，并制作程序化低多边形手持模型；模型默认隐藏，必须通过黑色方碑互动获得后才装备到 Alex 右手。
- 方碑获取逻辑：靠近出现 `E` 互动提示；按 `E` 打开对话界面，右侧对话对象显示 45 度透明背景黑色方碑立绘，缩放枪图片只出现在对话框内容区；按钮流程为先点“获取”获得工具，再点“继续”退出对话。
- 玩法采用“装备态 + 瞄准态 + 确认操作”三段式：按 `U` 装备/进入瞄准，屏幕中央出现缩放枪准星，锁定目标后按 `=` 放大、按 `-` 缩小。
- 缩放对象先尽量开放，但不缩放火星星球本身和天上的星星本身；其他星球上的物体、场景内物体都可以被缩放。
- 缩放效果有时间限制：每次缩放持续 1 分钟，时间结束后自动恢复到原始尺寸。
- 数值限制仍按相对原始尺寸计算，放大和缩小极限都是 3 倍：最大 `3x`，最小约 `0.33x`。
- 新增概念图 `assets/concepts/scale-gun-concept.png`，深色底、简洁流线型、适合宇航员手持。
- 新增黑色方碑绿幕源图 `assets/portraits/monolith-dialogue-green.png`，仅作为抠图源；运行时使用透明版 `assets/portraits/monolith-dialogue-transparent.webp` 作为方碑对话的右侧角色立绘。
- 方碑对话文案只做玩家可读的缩放枪介绍，不展示设计参数、效果时长、数值上限或“头盔自动展开图纸”等实现解释。
- 缩放枪 3D 模型已使用 Three.js 几何体拼装，材质沿用宇航服暖白、石墨黑、橙色安全件和蓝色镜头。

### 火星车轮迹与细雾粉尘

原因：

用户希望火星车驶过厚尘地表时留下车轮压痕，并在车尾生成细腻雾气型粉尘，要求不能做成大颗粒飞沙，且线上网站也要明显看得出效果。

决策：

- 第一版采用轻量视觉方案：贴地半透明轮迹 mesh + 低透明度软边缘 Sprite 粉尘。
- 轮迹由有限对象池循环复用，沿车辆两侧轮位生成，贴合弧面火星地表。
- 粉尘只在车尾和后轮附近生成，颜色接近火星尘土，透明度低，会扩散、变淡、消失，避免大颗粒视觉。
- `02 车辆 运输车` 调整为机器人车库周边低速巡逻，保证玩家进入基地附近就能看到轮迹和车尾粉尘。
- 暂不做实时地形变形或动态地表纹理，因为当前项目目标是网页端稳定运行，第一版应优先保证桌面和移动端性能。

### Elon支线任务与固定对白池

原因：

用户进一步明确：Elon当前虽然是实验型开放角色，但第一阶段仍要依靠固定知识库和大量固定对白互动。玩家需要在 `03 飞船 返回飞船` 上找到它，但 3 号飞船升降梯初始无法运行，必须先完成修复任务。

决策：

- Elon 在游戏内本名就是 `Elon`，只是同名，不指向现实中的 Elon Musk 本人，也不是数字复活。
- Elon 可被理解为接近马斯克式思维的工程思想人格或“精神体式”顾问，但不代表现实人物发言。
- Elon 的位置设在 `03 飞船 返回飞船` 升降平台到达顶端后的固定高空廊道，靠近飞船舱门处。
- `03 飞船 返回飞船` 内舱不可进入，玩家只在外部高空廊道靠近舱门处与 Elon 互动。
- 新增解锁支线：玩家需要寻找执行器驱动轴、高功率继电器、姿态锁止传感器、低温润滑胶囊，并在科研舱生成校准密钥，修复升降梯后才能见到 Elon。
- 升降梯修好后，玩家后续可以随时乘坐 3 号飞船升降梯，到达上层高空廊道与 Elon 交流。

### 桌面端常用键位向左手集中

原因：

玩家希望减少右手离开鼠标或左手离开 WASD 区域的次数。原头盔灯 `I` 和地图 `M` 都偏右，不适合单手连续操作；第一人称此前只能靠滚轮逐步缩近。

决策：

- 头盔灯从 `I` 改为 `R`。
- 地图从 `M` 改为 `F`，长按 `F` 进入全屏地图，右上角地图按钮同步显示 `F`。
- 新增 `C` 一键进入第一人称，相当于把相机直接缩到宇航员头部位置；退出第一人称仍可通过鼠标滚轮拉远。

### 缩放枪键位改为 X 与 Q/E

原因：

缩放枪原来用 `U` 开关瞄准、`=` 和 `-` 调整大小，键位偏右，不符合当前“左手集中操作”的方向。玩家希望瞄准后继续使用 `Q/E` 完成缩小和放大。

决策：

- 缩放枪开关瞄准从 `U` 改为 `X`，`U` 不再绑定缩放枪功能。
- 缩放枪瞄准态下，`E` 放大，`Q` 缩小。
- `=` 和 `-` 的缩放设定保留，作为备用操作。
- 瞄准态下 `Q/E` 优先控制缩放枪，不触发普通互动或二选项操作。

### 低氧时雷达高亮补氧点

原因：

氧气压力是当前 1.0 的核心生存反馈之一，但玩家低氧时只看到数值和文字警告，不一定知道最近的补氧点在哪里。任务目标已经有绿色放大点，补氧目标应该使用同一套雷达语言。

决策：

- 氧气背包低于或等于 35% 时，雷达开始高亮可补氧地点。
- 居住舱补给柜、氧气生产站和登陆飞船补给舱显示为放大的橙色发光点。
- 橙色补氧点的尺寸、发光和排序逻辑参考绿色任务目标，但任务目标仍保持更高优先级。
- 氧气高于 35% 时，补氧点不额外高亮，避免常态雷达过度拥挤。

### 体能改为居住舱恢复型生存指标

原因：

体能原来主要跟加速短时消耗和自动恢复绑定，玩家很难感知它是一个需要管理的生存指标。当前希望体能一般不容易耗完，但持续加速、连续跳跃会带来明显成本，并让居住舱承担“休整点”的作用。

决策：

- 地表普通移动会缓慢消耗体能。
- 持续加速和每次跳跃都会增加体能消耗。
- 体能低于或等于 20% 时提示回居住舱休整。
- 氧气补给点只补氧，不补体能。
- 进入或离开居住舱会补满氧气和体能。
- 体能耗尽时，Alex 在居住舱内重新同步，氧气和体能恢复满值。

### 出室内后临时使用人物正面镜头

原因：

从居住舱或飞船内舱出来时，普通跟随镜头会落在建筑或飞船主体一侧，导致建筑遮挡宇航员身体，玩家短时间看不清角色位置和移动方向。

决策：

- 从居住舱、温室或飞船内舱出来时，启动临时正面镜头。
- 临时镜头站在人物正前方看向 Alex，让主体建筑位于角色背后，不遮挡角色身体。
- 玩家离出口达到安全距离后，自动恢复普通第三人称跟随视角。
- 进入室内、重生或回标题时清除临时镜头状态。

### 09 医疗舱机器人外移到门外

原因：

09 机器人原来贴近医疗舱侧前方，容易被医疗舱模型或碰撞修正推到不易看见的位置，玩家到医疗舱时会误以为机器人被困在舱内。

决策：

- 09 机器人改到医疗舱正门外更开阔的巡逻点。
- 医疗舱模型可见编号从 06 统一改为 09。
- 医疗舱碰撞标签同步改为 09，避免后续维护时继续混淆编号。

### 视角复位和缩放枪倍率调整

原因：

`C` 只进入第一人称会让玩家不知道如何快速切回第三人称；`=` 和 `-` 调整镜头后也需要一个明确的默认视角复位键。缩放枪原来的 3 倍上限偏保守，玩家希望效果更明显。

决策：

- 缩放枪缩放范围从 `0.33x` 到 `3x` 调整为 `0.25x` 到 `4x`。
- `C` 改为第一人称和默认第三人称切换。
- 新增 `0` 键，一键恢复默认第三人称视角。
- 缩放枪瞄准态仍只用 `X` 循环打开/收起；`C` 和 `0` 不负责收起缩放枪。

### 体能加速跑消耗对齐跳跃

原因：

体能需要让玩家直觉理解不同动作的成本。跳跃是一次性扣除，加速跑是持续消耗，两者需要有明确换算关系。

决策：

- 一次跳跃消耗 `4` 点体能。
- 加速跑每秒消耗等于一次跳跃消耗，即每秒 `4` 点体能。
- 普通移动仍保持低消耗，避免玩家正常探索时频繁耗尽体能。

## 2026-06-25：多人 v1 采用免注册公共大厅

原因：

用户希望不同人进入游戏时自动生成新的宇航员，彼此可见，任务相同且不需要注册。当前 1.0 是纯前端静态网页游戏，因此多人能力需要新增实时同步层，但不应破坏现有单人可试玩闭环。

决策：

- 多人 v1 默认进入公共房间 `public-mars-alpha`。
- 浏览器首次进入生成匿名 `playerId`，保存在 `localStorage`；同一浏览器刷新后继续使用同一身份。
- 不使用 IP 作为玩家身份主键。IP 只可作为服务器日志、限流或防刷辅助，因为同一网络、VPN、校园网和公司网络都会造成误判。
- 任务内容相同，但进度各自独立；一个玩家完成任务不会推进其他玩家。
- 同步内容只包括远程宇航员的位置、朝向、移动速度、飞行状态、头盔灯状态和室内/室外状态。
- 远程玩家不可碰撞、不可互动，不参与任务触发。
- WebSocket 服务不可用时，前端安静降级为单人体验。

实现：

- 新增 `server/multiplayer.ts` 作为 Node WebSocket 房间转发服务。
- 新增 `src/multiplayer.ts` 作为前端多人客户端。
- 新增 `src/multiplayer-protocol.ts` 共享消息类型。
- `package.json` 新增 `dev:server` 和 `dev:multi`。

### 远处太阳改为实体球

原因：

原来的太阳使用 canvas 渐变 Sprite，远处看像一团朦胧光斑，不符合当前希望看到明确太阳球体的视觉要求。

决策：

- 太阳视觉从 `THREE.Sprite` 改为 `THREE.SphereGeometry` 实体球。
- 保留一层很薄的外发光壳用于亮度，但主体轮廓必须清楚。
- 风暴时只降低太阳实体和外发光透明度，不再回到大面积雾状光斑。
